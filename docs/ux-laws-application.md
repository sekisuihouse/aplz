# Laws of UX application notes

Source checked: https://lawsofux.com/ on 2026-06-18.

This note maps the current APLZ design changes to the Laws of UX list. It is intentionally implementation-oriented, not a full design essay.

| Law | Applied in APLZ |
| --- | --- |
| Aesthetic-Usability Effect | Cleaner spacing, stronger primary action, fewer competing borders. |
| Choice Overload | Header and search screens now expose fewer choices by default. |
| Chunking | Home flow, request form, and filters are grouped into small sections. |
| Cognitive Bias | Copy avoids claiming real user adoption; visible examples stay task-based. |
| Cognitive Load | Request posting asks for a title first and hides advanced details. |
| Doherty Threshold | Primary actions give immediate visual state through disabled/loading styles. |
| Fitts's Law | Main CTAs and form buttons use larger touch targets. |
| Flow | Home explains the write, clarify, try loop before secondary paths. |
| Goal-Gradient Effect | Request form shows 0/3 to 3/3 progress. |
| Hick's Law | Filters are collapsed behind one control; quick filters stay short. |
| Jakob's Law | Auth uses familiar sign-in/sign-up tabs and email-link language. |
| Law of Common Region | Search/filter and form sections use clear bounded regions. |
| Law of Proximity | Related labels, hints, and controls are placed together. |
| Law of Pragnanz | Pages favor simple pills, lists, and direct labels over decorative layout. |
| Law of Similarity | Repeated cards and pills share visual treatment. |
| Law of Uniform Connectedness | Numbered flows and progress indicators connect related steps. |
| Mental Model | APLZ is framed as: write a problem, answer questions, try an app. |
| Miller's Law | Primary navigation and top-level home choices stay under seven items. |
| Occam's Razor | Duplicative hero actions were removed or demoted. |
| Paradox of the Active User | Users can start with examples or a title without reading instructions. |
| Pareto Principle | The highest-value path, posting a problem, is the dominant CTA. |
| Parkinson's Law | The form sets a low bar: title-only submission is allowed. |
| Peak-End Rule | Posting screen keeps the final sticky action clear and reassuring. |
| Postel's Law | Request API and UI accept partial input while output stays structured. |
| Selective Attention | Blue accent is reserved for the main action and active states. |
| Serial Position Effect | Important actions appear first in hero/nav and last in sticky submit. |
| Tesler's Law | Necessary complexity remains in optional details instead of being removed. |
| Von Restorff Effect | The primary CTA visually differs from secondary actions. |
| Working Memory | The request form shows progress and short hints instead of long instructions. |
| Zeigarnik Effect | Incomplete form steps are visible as light prompts, not blockers. |
